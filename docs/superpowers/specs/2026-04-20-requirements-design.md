## 4. 功能需求

### 4.1 策略管理功能

#### FR-1: 更新策略信息

**需求描述：** 用户可以修改策略的部分信息

**用户故事：**
> 作为量化投资者，我希望能够更新策略的描述和风险等级，以便反映策略的最新情况

**功能规格：**
- **可修改字段：** description（描述）、risk_level（风险等级）
- **不可修改字段：** name（策略名称）、initial_capital（初始资金）
- **系统计算字段：** total_return、max_drawdown、sharpe_ratio（不接受手动输入）

**输入验证：**
- description：最大 500 字符
- risk_level：枚举值 ["低", "中低", "中", "中高", "高"]

**输出：**
- 成功：返回更新后的策略信息
- 失败：返回错误信息和原因

#### FR-2: 删除策略

**需求描述：** 用户可以删除不需要的策略

**用户故事：**
> 作为量化投资者，我希望能够删除废弃的策略，以便保持系统整洁

**功能规格：**
- 采用**软删除**方式（标记为 deleted，不删除物理数据）
- 删除后，查询时不显示该策略
- 策略的关联持仓和交易记录保留
- 记录删除审计日志

**安全考虑：**
- 删除前检查是否有关联持仓
- 提示用户："策略下有 X 个持仓，确认删除？"

### 4.2 交易记录管理功能

#### FR-3: 修改交易记录

**需求描述：** 用户可以修正录入错误的交易记录

**用户故事：**
> 作为量化投资者，我希望能够修正录入错误的交易价格或数量，以便保证数据准确性

**功能规格：**
- **可修改字段：** price（价格）、quantity（数量）、fee（手续费）、notes（备注）
- **必填字段：** reason（修改原因）

**输入验证：**
- price：必须 > 0
- quantity：必须 > 0
- fee：必须 >= 0
- reason：必填，最大 200 字符

**审计要求：**
- 记录修改前的值和修改后的值
- 记录修改原因和修改时间
- 支持查询审计历史

**业务规则：**
- 修改后自动重新计算持仓的市值和盈亏
- 修改后更新策略的资金状况
- **标记策略为 dirty 状态**，触发异步重算任务
- 重算任务修复所有衍生数据（净值曲线、收益率等）

**修改约束（推荐方案）：**
- 允许修改历史交易（不限制时间范围）
- 但系统将标记对应 strategy 为 dirty 状态
- dirty 策略的数据可能暂时不准确
- 后续通过后台重算任务自动修复所有衍生数据
- 前端显示 dirty 状态提示（如"数据更新中..."）

**数据一致性保证：**
- 交易修改 → 触发重算 → 标记 dirty → 后台任务修复 → 清除 dirty 状态
- 重算期间可以正常查询，但数据可能暂时是旧值
- 重算完成后自动更新为最新值

### 4.4 数据重算机制

#### FR-8: 数据重算执行策略

**需求描述：** 定义数据重算的执行策略，确保系统在数据变更后的一致性

**用户故事：**
> 作为量化投资者，我希望系统能够自动维护数据一致性，即使我修改了历史交易，所有相关数据也能正确更新

**重算范围与策略：**

采用**全量重算策略**（Phase 2 简化方案）：

- **重算粒度**：以 strategy_id 为单位
- **触发重算时**，对该 strategy 下所有数据进行全量重算
- **重算内容包括**：
  - 持仓数量与成本
  - 当前市值（market_value）
  - 总资产（total_assets）
  - 盈亏（profit_loss / profit_loss_pct）
  - 净值曲线数据（daily_portfolio_summary）

**说明：**
- Phase 2 不实现增量重算（复杂度较高）
- 后续如有性能瓶颈，可演进为增量重算

**数据一致性分级：**

| 数据类型 | 一致性要求 | 说明 |
|----------|------------|------|
| 持仓数量 | 强一致（不允许脏数据） | 核心交易数据 |
| 当前市值 | 最终一致（允许短暂延迟） | 分析类数据 |
| 总资产 | 最终一致 | 允许短暂延迟 |
| 今日盈亏 | 最终一致 | 允许短暂延迟 |
| 净值曲线 | 最终一致 | 允许短暂延迟 |

**要求：**
- 核心交易数据（positions）必须始终准确
- 分析类数据允许在重算期间短暂不一致

**dirty 状态生命周期：**

定义 strategy 的状态机：

```
clean（数据一致）
  ↓ 数据变更
dirty（数据已变更，等待重算）
  ↓ 重算任务触发
recomputing（正在重算）
  ↓ 重算成功/失败
clean / failed
```

**状态流转规则：**
- `dirty → recomputing → clean`：正常重算流程
- `recomputing → failed`：重算失败，保留 dirty 状态
- `failed → recomputing`：允许手动或自动重试

**异常处理机制：**

- **重算失败时**：
  - 标记策略为 failed 状态
  - 保留 dirty 状态
  - 记录失败原因到审计日志
  - 允许手动触发重算
  - 后台任务自动重试（最多3次）

- **防止重复触发**：
  - 若状态为 recomputing，忽略新的重算请求
  - 或合并重算请求（标记为 pending）

**重算触发时机：**
- 交易记录创建/修改/删除
- 持仓创建/修改/删除
- 价格更新（手动触发）
- 策略资金调整

### 4.5 持仓计算规则

#### FR-9: 持仓计算规则定义

**需求描述：** 明确持仓计算的业务规则，确保数据一致性

**用户故事：**
> 作为量化投资者，我希望系统能够按照统一的规则计算持仓成本和盈亏，以便准确评估投资表现

**成本计算方法：**

采用**加权平均成本法**（Weighted Average Cost）：

- **买入时**：
  ```
  新成本 = (原持仓成本 × 原数量 + 新交易金额) / (原数量 + 新数量)
  ```

- **卖出时**：
  ```
  成本保持不变（按平均成本减少持仓）
  实现盈亏 = 卖出价 - 成本价 × 卖出数量
  ```

**持仓数量计算：**

```
持仓数量 = 所有买入数量 - 所有卖出数量
```

**不允许出现负持仓**（如出现需视为数据异常）

**盈亏计算规则：**

- **未实现盈亏**：
  ```
  未实现盈亏 = (当前价 - 持仓成本) × 当前持仓数量
  ```

- **已实现盈亏**：
  ```
  已实现盈亏 = (卖出价 - 成本价) × 卖出数量
  ```

**交易修改影响规则：**

当交易记录被修改时：

- **必须从该策略的所有交易重新计算**：
  - 持仓数量
  - 持仓成本
  - 盈亏数据
- **不允许局部修补**（避免累计误差）

**计算示例：**

```
示例：持有平安银行
1. 初始买入：1000股 @ 10.00元，成本 = 10,000元
2. 加仓：500股 @ 12.00元，金额 = 6,000元
   新成本 = (10,000 + 6,000) / (1000 + 500) = 10.67元
3. 部分卖出：800股 @ 11.00元
   实现盈亏 = (11.00 - 10.67) × 800 = 264元
4. 剩余持仓：700股，成本 = 10.67元
```

**未来扩展说明：**
- 当前阶段仅支持加权平均成本法
- 后续可扩展：FIFO（先进先出）、LIFO（后进先出）、多账户独立成本核算

  },
  "charts": [
    {
      "id": "position_distribution",
      "type": "pie",
      "metric": "distribution",
      "title": "持仓分布",
      "enabled": true,
      "options": {
        "dataSource": "/api/analytics/portfolio/distribution",
        "labelField": "name",
        "valueField": "market_value",
        "showPercentage": true
      }
    },
    {
      "id": "nav_curve",
      "type": "line",
      "metric": "nav",

  - `enabled`：是否启用
  - `options`：图表特定选项

**向后兼容：**
- 所有配置变更必须保证向后兼容
- 新增字段需有默认值
- version 字段用于检测配置格式变更

## 6. 数据需求

### 6.1 数据一致性模型

为确保系统在交易修改、策略更新等操作后的数据一致性，定义如下原则：

**冗余存储 + 衍生更新模式：**
- 持仓数据（positions）、策略指标（strategies）采用冗余存储 + 衍生更新模式
- 不采用纯实时计算（性能考虑）
- 所有影响持仓的数据变更必须触发数据重算流程

**数据重算范围：**
- 默认：仅影响对应的 strategy_id
- 重算内容包括：
  - 持仓数量
  - 市值（market_value）
  - 总资产（total_assets）
  - 盈亏（profit_loss, profit_loss_pct）
  - 策略净值曲线

**重算触发时机：**
- 交易记录创建/修改/删除
- 持仓创建/修改/删除
- 价格更新（手动触发）
- 策略资金调整

### 6.2 数据存储需求

**新增数据表：**
1. `transaction_audit_log` - 交易审计日志
2. `strategy_audit_log` - 策略审计日志
3. `daily_profit_loss` - 每日盈亏快照
4. `daily_portfolio_summary` - 每日组合汇总
5. `chart_configs` - 图表配置
6. `chart_templates` - 图表模板

**数据保留策略：**
- 审计日志：永久保留
- 每日快照：保留 5 年
- 交易记录：永久保留

### 6.2 数据准确性需求

**数据验证：**
- 所有输入数据必须验证
- 数值类型检查
- 范围检查
- 格式检查

**数据一致性：**
- 使用数据库事务
- 外键约束
- 级联更新（软删除）
